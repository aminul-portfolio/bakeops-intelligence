BI-ready reporting layer

Exports and reviewer walkthrough

BakeOps V1 should produce stable CSV outputs so dashboard metrics can be inspected outside the app, tested against source records, and reused in BI tools.

Required V1 exports

BI-ready CSV files

These files should be generated from the same data and metric logic used by the dashboard.

Export file Purpose Grain
fact_orders.csv Paid order revenue and status context One row per order
fact_order_items.csv Product quantity, line total, and item-level demand One row per order item
fact_waste.csv Waste reason, ingredient/product context, and estimated cost One row per waste record
fact_production_batches.csv Production batch output and allocation context One row per batch line/allocation
dim_cake.csv Product and category attributes One row per cake or variant
dim_ingredient.csv Ingredient, supplier, cost, stock, and reorder attributes One row per ingredient/lot
dim_customer.csv Customer and loyalty reporting dimension One row per customer
daily_bakery_metrics.csv Gold-layer KPI snapshot for dashboard validation One row per workspace per day
product_performance_snapshot.csv Revenue, cost, margin, waste-adjusted margin, ratings One row per product/variant/period
Command target

Export command

python manage.py export_bi_csv

This command can be optional in early V1, but the export service should use the same metric definitions as the dashboard to avoid reporting drift.

Reviewer proof

What exports prove

  • Dashboard values are not isolated UI numbers.
  • Metrics can be reused by BI/reporting tools.
  • Operational data is transformed into a reporting layer.
  • Export totals can be tested against dashboard totals.
Portfolio note

V1 scope boundary

V1 focuses on CSV exports. JSON, Excel, and Parquet can wait until V3 when the platform moves toward commercial SaaS readiness.
