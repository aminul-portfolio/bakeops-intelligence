# BakeOps Intelligence - Metric Governance

## Purpose

This document explains how BakeOps Intelligence builds, stores, verifies, and protects its analytics metrics.

The goal is to show reviewers that the analytics pages are not hard-coded screenshots or presentation-only values. They are based on operational records, a repeatable metric build command, stored gold-layer snapshots, data quality checks, and exportable BI files.

---

## Governance Principle

BakeOps separates operational data from reporting metrics.

```text
operational bakery records
-> metric build command
-> gold-layer metric snapshots
-> analytics pages
-> BI export files
```

The dashboard is not the source of truth. The source of truth is the stored data model plus the repeatable build process.

---

## Metric Build Command

Metrics are generated by:

```powershell
python manage.py build_bakery_metrics
```

Expected current output pattern:

```text
BakeOps bakery metrics built successfully.
Workspace: SweetCakes Bakery
Rows processed: 12
Metrics created: 1
Snapshots created: 21
Issues created: 12
```

The command is designed to be repeatable after seeding demo data.

---

## Gold-Layer Outputs

The build command creates or refreshes these trusted reporting outputs:

| Output | Purpose |
|---|---|
| `DailyBakeryMetric` | Daily KPI summary |
| `ProductPerformanceSnapshot` | Product-level revenue, margin, waste-adjusted margin, rank movement, and action flag |
| `IngredientUsageSnapshot` | Ingredient stock risk, expiry pressure, waste cost, and reorder signal |
| `OccasionDemandSnapshot` | Occasion demand, revenue contribution, upcoming order count, and pressure level |
| `CustomerLoyaltySnapshot` | Customer revenue, order count, average order value, loyalty points, and repeat status |
| `DataQualityIssue` | Operational trust issues that need review |
| `BakeryMetricRunLog` | Audit trail for metric build runs |

## Signature Metric Governance

The key reviewer proof is:

```text
Birthday Classic 1 4 review
```

Meaning:

| Field | Value |
|---|---|
| Product | Birthday Classic |
| Revenue rank | 1 |
| Waste-adjusted margin rank | 4 |
| Action flag | review |

This metric matters because it proves the platform is not only reporting revenue. It also evaluates product strength after ingredient cost and waste impact are included.

## Data Quality Governance

BakeOps does not hide operational data issues.

The metric build creates `DataQualityIssue` records so reviewers can see that the platform includes a trust layer, not only positive KPI cards.

Current expected evidence:

```text
Issues created: 12
```

The data quality page is available at:

```text
/analytics/data-quality/
```

---

## Run Log Governance

`BakeryMetricRunLog` provides build-level audit evidence.

It supports reviewer questions such as:

- did the metric build run?
- how many rows were processed?
- how many metrics were created?
- how many snapshots were created?
- how many issues were created?
- did the command finish successfully?
- when did the run start and finish?

This gives the platform a stronger AE-style governance story than a simple dashboard-only project.

---

## Export Governance

BI exports are generated by:

```powershell
python manage.py export_bi_csv
```

Expected current evidence:

```text
Files generated: 11
Total rows exported: 52
```

The export layer allows the same modelled data to be reviewed outside Django in tools such as Excel, Power BI, or other BI workflows.

Generated exports are reproducible and should not be committed, except for `exports/.gitkeep`.

## Current Boundary

BakeOps can honestly claim:

- repeatable seeded demo data
- repeatable metric build
- stored gold-layer snapshots
- data quality issue generation
- run log evidence
- trusted analytics pages
- BI-ready exports
- tests and CI workflow

BakeOps must not claim:

- live customer usage
- production deployment
- paid subscriptions
- Stripe billing
- POS integration
- Shopify/Square integration
- automated customer onboarding

unless those features are genuinely implemented and verified.
