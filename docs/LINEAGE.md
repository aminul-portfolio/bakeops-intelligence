# BakeOps Intelligence - Data Lineage

## Purpose

This document explains how data moves through BakeOps Intelligence from seeded operational records to trusted analytics pages and BI-ready CSV exports.

It is designed for reviewers who want to understand:

- where the data starts
- how the metric layer is built
- which models support the analytics pages
- how the dashboard connects to exported BI files
- what is implemented now versus planned for future commercial work

---

## End-to-End Lineage

```text
seed_demo_data
-> operational bakery records
-> build_bakery_metrics
-> gold-layer metric snapshots
-> trusted analytics pages
-> export_bi_csv
-> BI-ready CSV files
```

The dashboard is not treated as the source of truth. BakeOps stores operational records first, then builds gold-layer reporting snapshots that power analytics pages and exports.

---

## Source Layer

Current source data is seeded demo data created by:

```powershell
python manage.py seed_demo_data --reset
```

The seeded demo creates a controlled bakery workspace with:

```text
Workspace: SweetCakes Bakery
Cakes: 4
Orders: 7
Ingredients: 8
Waste records: 4
```

This is intentionally demo data. BakeOps V3 documents future import readiness, but live external imports are not implemented yet.

## Operational Layer

Operational records represent the business events and reference data used for metric calculation.

| Model | Role |
|---|---|
| `Workspace` | Bakery workspace boundary |
| `Customer` | Customer profile data |
| `LoyaltyAccount` | Loyalty and repeat-customer context |
| `CakeReview` | Customer review evidence |
| `OccasionType` | Occasion classification |
| `DeliverySlot` | Delivery and fulfilment timing |
| `BakeryOrder` | Order header |
| `BakeryOrderItem` | Order line items |
| `Supplier` | Ingredient supplier information |
| `Ingredient` | Ingredient master data |
| `IngredientLot` | Stock, expiry, and cost information |
| `Recipe` | Product recipe structure |
| `RecipeLine` | Ingredient quantity per recipe |
| `ProductionBatch` | Production batch event |
| `ProductionBatchLine` | Batch output by cake |
| `BatchAllocation` | Batch-to-order allocation |
| `WasteRecord` | Waste event and reason |

## Metric Build Layer

Metrics are built by:

```powershell
python manage.py build_bakery_metrics
```

The build command processes operational records and creates stored analytics snapshots.

Expected current evidence:

```text
Rows processed: 12
Metrics created: 1
Snapshots created: 21
Issues created: 12
```

---

## Gold-Layer Analytics Layer

Gold-layer records are the reporting-ready outputs used by analytics pages and exports.

| Model | Purpose |
|---|---|
| `DailyBakeryMetric` | Daily KPI summary |
| `ProductPerformanceSnapshot` | Product revenue, margin, waste-adjusted margin, ranking, and action flag |
| `IngredientUsageSnapshot` | Ingredient usage, waste, stock risk, and expiry pressure |
| `OccasionDemandSnapshot` | Occasion demand, revenue, upcoming orders, and delivery pressure |
| `CustomerLoyaltySnapshot` | Customer revenue, order count, AOV, loyalty points, and repeat status |
| `DataQualityIssue` | Data trust issues created during metric build |
| `BakeryMetricRunLog` | Build audit trail with status, row counts, and timing |

## BI Export Layer

BI-ready CSV files are generated by:

```powershell
python manage.py export_bi_csv
```

Expected current evidence:

```text
Files generated: 11
Total rows exported: 52
```

## Signature Insight Lineage

BakeOps proves this business idea:

```text
A best-selling bakery product can become weak after ingredient cost and waste are included.
```

The seeded demo proves it with:

```text
Birthday Classic 1 4 review
```

Meaning:

| Product | Revenue Rank | Waste-adjusted Margin Rank | Action |
|---|---:|---:|---|
| Birthday Classic | 1 | 4 | review |

This proves that the highest-revenue product is not necessarily the strongest product after waste-adjusted profitability is calculated.

## Current Boundary

Implemented now:

- seeded operational demo data
- repeatable metric build command
- gold-layer snapshot records
- trusted analytics pages
- data quality issue generation
- BI-ready CSV exports
- tests and CI workflow

Not implemented yet:

- live external POS import
- Shopify/Square integration
- automated customer onboarding
- billing or subscriptions
- production SaaS deployment
- real customer usage

These are future concerns and must not be claimed as implemented.
